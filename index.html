01 / The problem

One design.
For everyone.

Every app ships with a fixed layout. A grid someone else chose. A sidebar nobody asked for. The developer decides what you see and how it's organized, not because they know you, but because there was no other way.

Until now, that constraint was technical. It no longer is.

Traditional software

Interface as assumption

The designer imagined an "average user" and built for them. You adapt. You learn which menu hides the thing you need. You tolerate the layout.

N/A

Interface as output

You describe what you want. The UI rebuilds itself around your mental model, not just the theme, the actual structure, layout, and components.

Traditional software

Customization as cosmetics

Dark mode. Font size. Maybe a color theme. The bones never change. It's still the same product; you just repainted it.

N/A

Every user is the designer

The interface you're using was generated for you, from a conversation. No two sessions look alike, because no two people think alike.

02 / The demo

An email client
as proof of concept.

To show what N/A makes possible, we built a prototype email client on top of it. The inbox is the demo. Start from a baseline, describe what you want, and watch the interface rebuild itself.

01

Natural language → UI

Describe what you want in plain English. A new component is generated and mounted, layout, data bindings, and all.

02

Inspect mode

Click any element. Describe what you want changed. Only that part updates, surgical edits without rewriting everything around it.

03

Version history

Every AI change is a restore point. Navigate backward and forward through your interface's history the way you'd use git.

04

Shareable snapshots

Any view can be shared via URL. The recipient gets an isolated copy, not your live session, not a screenshot. A working fork.

05

Persona system

Different user types start from different defaults. A manager, a founder, a support lead. Each begins somewhere different and diverges further from there.

06

No build step

The interface runs in the browser. Changes take effect immediately. There's no compile-deploy cycle between "I want this" and "I have it."

03 / The vision

What this means
for software.

YC's current Request for Startups calls out dynamic software interfaces as one of the most underexplored consequences of capable AI. We think they're right, but the framing undersells what's actually possible.

The fixed interface isn't just inefficient. It's a category error. We're shipping static documents and calling them products. We decided, decades ago, that software would be authored once and consumed by many, because that was the only viable economic model. That constraint is gone.

The interface you use shouldn't look like the interface I use. We don't think alike. We don't work alike. We shouldn't be forced to use alike.

N/A is a demo of what happens when you take this seriously: not "AI-assisted configuration," but the interface as a generated artifact, one that's produced fresh for each user, shaped by what they ask for, and mutable at any moment.

Email is the right proving ground because it's old, everyone uses it, and everyone has opinions about how it should work. The inbox you want doesn't exist. So we built a system that lets you describe it, as a way of showing what N/A can do.

N/A itself is a developer tool. The goal is to give builders a runtime where interface malleability is a first-class primitive, not an afterthought bolted onto a fixed layout. Eventually, a platform where apps built on N/A can share components, personas, and snapshots across users.

If this works for email, it works for everything.
